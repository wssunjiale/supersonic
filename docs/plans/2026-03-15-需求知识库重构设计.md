# 需求知识库重构设计

## 背景

当前 `pr/` 目录虽然已经从“原始聊天摘录”提升到了“阶段/单元归档”，但仍然没有真正以“需求”为主线组织信息，主要问题有四类：

- `bug` 仍被当成并列主题，而不是挂在具体需求下的修复记录。
- 需求缺少“是否仍然有效”的时效标记，后续需求覆盖前述需求时，旧需求不会自动退出主线。
- 归档单元仍然偏向“聊天主题”，而不是“真实、独立、可追踪、边界清晰、可测试的需求”。
- `pr/` 目录中仍存在英文文件名和英文目录名，不满足“全部中文”的要求。

## 目标

将 `pr/` 目录重构为“以需求为唯一主线”的知识库：

- 先识别真实需求，再把相关 `bug`、修复、约束、决策、验收挂在该需求下面。
- 明确区分“当前有效需求”和“已过时需求”。
- 允许“已实现但仍然有效”的需求继续保留在主线中，而不是误标成“已过时”。
- 所有生成文件与目录都使用中文命名。

## 核心模型

### 1. 需求

需求是知识库中的最小主线单位，必须满足：

- 真实：来自用户明确提出或多轮对话收敛后的真实意图。
- 明确：可以用一句话准确描述“到底要做什么”。
- 独立：可以独立理解，不依赖阅读整段聊天。
- 边界清晰：有明确的边界、非目标、限制条件。
- 可追踪：能回溯来源会话与关键决策。
- 可测试：可以写出明确的验收标准。

### 2. 问题与修复记录

`bug` 不再单独作为需求存在，而是作为“需求下的关联问题与修复记录”存在。每条记录至少包含：

- 现象
- 根因
- 修复/处理方式
- 影响范围
- 来源会话

### 3. 需求状态

每个需求包含两套状态：

- 时效状态：`有效` / `已过时`
- 实现状态：`未判定` / `进行中` / `已落地`

解释规则：

- “已实现但仍然是现行规则”的需求，时效状态仍然是 `有效`。
- 只有“被后续需求明确覆盖、替代或否定”的需求，才标记为 `已过时`。

### 4. 覆盖关系

需求之间允许存在：

- `替代`：新需求覆盖旧需求。
- `依赖`：某需求依赖另一需求先成立。
- `关联`：两个需求相关，但不存在覆盖关系。

## 文档结构

`pr/` 目录改为以下结构：

- `说明.md`
- `00-项目基线.md`
- `01-当前有效需求索引.md`
- `02-已实现需求索引.md`
- `03-已过时需求索引.md`
- `04-问题与修复索引.md`
- `05-补充信息.md`
- `06-需求关系图.md`
- `需求/*.md`
- `知识库.json`
- `工具/生成需求知识库.py`

## 需求卡片结构

每个需求卡片固定包含：

- 需求标题
- 需求编号
- 需求状态
  - 时效状态
  - 实现状态
- 所属阶段
- 需求描述
- 业务目标
- 边界与非目标
- 已确认决策
- 验收标准
- 关联问题与修复记录
- 覆盖/替代关系
- 相关项目/模块
- 来源会话

## 生成策略

生成器采用“规则驱动 + 结构化抽取”的方式：

1. 解析聊天会话，按项目归档。
2. 先识别需求，不先识别 `bug`。
3. 同一需求的多轮会话归并到同一张需求卡片。
4. 对包含异常、报错、排障、修复内容的会话，抽取为需求下的问题与修复记录。
5. 对“被覆盖/被替代”的旧需求，标记 `已过时`，并明确指向替代需求。
6. 生成新的中文文件结构，并清理旧的英文文件与旧结构。

## 设计取舍

### 方案选择

本次采用“需求中心化重构”方案，而不是在现有“阶段/单元”结构上做局部补丁，原因是：

- 用户后续检索的目标是“找当前该做的需求”，不是“回忆某次聊天聊了什么”。
- `bug` 与修复经验的语义从属于需求，不应与需求并列。
- 只有引入“时效状态”后，知识库才能避免被历史废弃需求污染。

### 仍然保留的内容

- 阶段仍然保留，但仅作为需求的元数据，不再作为主索引中心。
- 跨项目关系、外部问题解决方法、长期偏好仍继续保留。

## 验收标准

- `pr/` 下不再出现英文文件名与英文目录名。
- 索引文件不再以“阶段 -> 单元”为主，而是以“需求”为主。
- `bug` 不再作为并列需求存在，而是进入具体需求卡片的“关联问题与修复记录”。
- 存在明确的“已过时需求索引”，且至少能正确表达被覆盖关系。
- 随机打开一张需求卡片时，能直接看出：
  - 这是一个什么需求
  - 现在是否仍然有效
  - 是否已经落地
  - 遇到过哪些问题
  - 已有哪些修复结论
