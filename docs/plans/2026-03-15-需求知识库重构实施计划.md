# 需求知识库重构实施计划

> **给后续执行者：** 本计划用于把 `pr/` 目录重构为“需求中心化知识库”，执行时应按步骤逐项完成，不要跳步。

**目标：** 将 `pr/` 目录从“阶段/单元归档”重构为“需求主线 + 问题修复挂靠 + 时效标记”的知识库。

**总体思路：** 重写生成器，先识别需求，再把 `bug`/修复记录挂到需求下；同时引入“有效/已过时”与“未判定/进行中/已落地”两套状态，并统一改为中文文件结构。

**技术栈：** Python 3、Markdown、JSON、规则驱动抽取

---

### 任务 1：固化新的数据模型

**文件：**

- 新建：`pr/工具/生成需求知识库.py`
- 参考：`docs/plans/2026-03-15-需求知识库重构设计.md`

**步骤：**

1. 定义“阶段”与“需求”的配置模型。
2. 在需求定义里加入：
   - 需求编号
   - 中文文件名
   - 时效状态
   - 实现状态
   - 替代关系
   - 依赖关系
3. 定义“问题与修复记录”的抽取结构。

### 任务 2：重写会话归并逻辑

**文件：**

- 修改：`pr/工具/生成需求知识库.py`

**步骤：**

1. 保留现有的项目识别、路径脱敏和消息清洗能力。
2. 改为“先判定需求，再判定该会话是否包含问题/修复记录”。
3. 不再生成独立的 bug 单元。
4. 对明显是流程性噪声、技能切换、系统会话、外部分析会话的记录继续过滤或分流。

### 任务 3：重写输出结构

**文件：**

- 生成：
  - `pr/说明.md`
  - `pr/00-项目基线.md`
  - `pr/01-当前有效需求索引.md`
  - `pr/02-已实现需求索引.md`
  - `pr/03-已过时需求索引.md`
  - `pr/04-问题与修复索引.md`
  - `pr/05-补充信息.md`
  - `pr/06-需求关系图.md`
  - `pr/需求/*.md`
  - `pr/知识库.json`

**步骤：**

1. 按需求生成索引，而不是按会话或单元生成索引。
2. 将“已实现但仍有效”的需求同时保留在有效需求主线中。
3. 将“已过时”需求单独输出，并明确其被哪个需求替代。
4. 生成问题与修复索引时，按“需求 -> 问题记录”组织。

### 任务 4：清理旧结构

**文件：**

- 删除旧的：
  - `pr/README.md`
  - `pr/01-需求索引.md`
  - `pr/02-问题解决方法.md`
  - `pr/03-补充信息.md`
  - `pr/04-验收与验证口径.md`
  - `pr/knowledge-base.json`
  - `pr/units/*`
  - `pr/tools/build_codex_knowledge.py`

**步骤：**

1. 由新生成器在生成前清理旧结构。
2. 清理后保证 `pr/` 中只保留中文目录与中文文件名。

### 任务 5：优先校验 Supersonic

**文件：**

- 检查：
  - `pr/01-当前有效需求索引.md`
  - `pr/03-已过时需求索引.md`
  - `pr/04-问题与修复索引.md`
  - `pr/需求/*.md`

**步骤：**

1. 确认 `supersonic` 中旧的“语义模型同步到 Superset dataset”已被标为 `已过时`。
2. 确认 `bug` 没有再以独立需求出现。
3. 抽查至少 3 张需求卡片，确认边界、验收、问题修复记录齐全。
