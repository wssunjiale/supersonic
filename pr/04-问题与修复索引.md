# Supersonic 问题与修复索引

这里只保留“已经明确归属于某个需求”的问题记录。

## [明确构建、测试与独立运行入口](需求/明确构建测试与独立运行入口.md)

- 子模块直接跑测试经常缺依赖：应以仓库根目录构建和测试为主。
- `Run Java` 找不到 `StandaloneLauncher`：本质是仓库导入方式、JDK 21 与 Maven 模型识别问题。

## [建立 Superset 双认证连接策略](需求/建立Superset双认证连接策略.md)

- JWT 登录失败后没有稳定回退到 API Key：需要明确优先级、回退条件和配置项。
- 消息体嵌入时访问 `/api/v1/me/roles/` 返回 401：说明嵌入认证链路并没有真正打通。
- JWT 和 API Key 配置读取不一致：会导致“回退后仍然不可用”的假回退。

## [完善嵌入式图表展示与交互](需求/完善嵌入式图表展示与交互.md)

- `superset-embed-test.html` 需要自动获取 token 和嵌入元信息，不能依赖手填。
- 嵌入区域过小、自适应差：需要同时考虑消息容器和 iframe 内容高度。

## [保证消息管理页面可用](需求/保证消息管理页面可用.md)

- 后端已经返回消息数据，但前端页面为空：根因是没有按 `ResultData` 包装结构解包。

## [将推送入口改为推送到看板](需求/将推送入口改为推送到看板.md)

- 推送目标列表为空：测试页和聊天页的看板列表链路没有统一。
- 自动生成的单图临时看板混入目标列表：需要从“可推送看板”中过滤。

## [完善图表类型选择与 form_data 生成规则](需求/完善图表类型选择与form_data生成规则.md)

- `form_data` 引用的字段与 dataset 真实列不一致：字段名优先级没有收敛。
- `SupersetVizTypeSelector` 在缺少模型时静默退化：需要显式失败或可解释兜底。
- 现有 `viztype` 资产是否足够：必须继续人工核查，不能假设已经可直接生成全部参数。

## [按查询结果固化为 Superset 数据集](需求/按查询结果固化为Superset数据集.md)

- 早期把“语义模型”误当成 Superset dataset：这是被后续需求明确否定的旧认知。
- 数据集复用、命名、tag、物理 / 虚拟判定，都要基于最终 SQL 与最终输出列来做。

## [建立 Superset 语义同步链路与离线验收运行器](需求/建立Superset语义同步链路与离线验收runner.md)

- 错误 SQL 不应继续同步到 Superset。
- 自动重试必须区分“可恢复故障”和“内部逻辑错误”。
- 日期格式 422、`already exists`、sync 500 等兼容问题，都属于同一条同步治理主线。

## [重构 Text2SQL 的 Schema Mapping 与 few-shot 策略](需求/重构Text2SQL的SchemaMapping与few-shot策略.md)

- LLM 更容易命中 `*_id` / `*_key` 等技术字段，而不是用户真正要看的可读字段。
- 字段别名、展示名、物理列名在多轮改写中被混淆，导致 SQL 错列或错聚合。
- few-shot exemplar 取样和候选字段暴露方式本身存在边界 bug。

## [提升语义查询链路稳定性](需求/提升语义查询链路稳定性.md)

- `parseInfo`、维度响应等对象为空时直接空指针。
- 字段名含空格、脏数据或错误请求体，会直接打爆 SQL 解析或反序列化链路。
- CLI 用的 Noop Bean 被 Web 服务错误注入，导致运行时抛 `UnsupportedOperationException`。

## [改善查询空结果时的回答质量](需求/改善查询空结果时的回答质量.md)

- 查询结果为空时，结果总结仍套用普通数据分析模板，最终把占位符原样输出给用户。
