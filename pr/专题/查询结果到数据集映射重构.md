- [建立 Superset 语义同步链路与离线验收运行器](../需求/建立Superset语义同步链路与离线验收runner.md)
- [基于 Headless 数据集同步链路重构聊天绘图与 Superset 嵌入方案](../需求/基于headlessdataset同步链路重构聊天绘图与Superset嵌入方案.md)

## 这个专题解决什么认知问题

早期实现里，容易把 “Supersonic 语义模型 / 数据集” 直接等同于 “Superset dataset”。  
用户在 `2026-02-05` 的会话里明确否定了这种理解，并给出了新的主线：

- Superset chart 只能依赖单个 dataset；
- Supersonic 的一次语义查询可能跨多个语义模型；
- 因此真正应该映射到 Superset dataset 的，不是语义模型本身，而是“一次查询最终形成的结果结构”。

## 重构后的核心结论

- 一个 Superset dataset 对应的应该是：
  - 一次查询产生的最终 SQL
  - 这次查询最终暴露出的输出列
  - 这次查询的指标 / 维度 / 过滤组合
- 复用判定可以先采用“SQL 规范化 hash”这类简单规则，再逐步补充命名、说明、tag 和生命周期治理。
- 如果最终 SQL 只落到一张物理表，可以优先按物理 dataset 处理；否则按虚拟 dataset 处理。

## 这项认知为什么重要

- 它直接导致 `需求-11` 失效，并被 `需求-12` 取代。
- 它决定了 `需求-13` 里同步治理链路应该管理什么对象。
- 它也是 `需求-21` 能否成立的前提：聊天绘图链路必须建立在“正确的数据集定义”之上。

## 仍需持续关注的实现点

- 数据集命名、说明、tag 应该由 Supersonic 统一管理，而不是让 Superset 中散落出不可治理的匿名 dataset。
- 当不同查询可复用同一 dataset 时，不应重复创建定义。
- 错误 SQL 或不完整结果不应被固化为 dataset，这属于 `需求-13` 的治理边界。
