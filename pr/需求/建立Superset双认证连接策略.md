# 建立 Superset 双认证连接策略

- 需求主线：`Superset 认证与连接`

## 当前定义

- Superset 接入同时支持 JWT 和 API Key 两条认证路径，并能按配置决定优先级，在首选路径失败时自动回退。

## 用户要求

- 优先尝试 JWT；如果 JWT 可用，就不走 API Key。
- JWT 失败时要自动回退到 API Key，而不是直接终止。
- 优先级需要可配置，而不是写死在代码里。
- 消息体嵌入和测试页嵌入都必须真正打通认证，不接受“测试页可以，正式消息不行”。

## 需求边界

- 本需求解决“如何稳定连上 Superset 并拿到有效认证”。

## 验收要点

- JWT 与 API Key 都可以配置。
- 首选认证失败时，系统会自动进入回退路径。
- 消息体内嵌与测试页嵌入都能基于正确认证稳定访问 Superset。

## 归属的 bug / 修复

### 问题 1：JWT 失败后没有稳定回退

- 现象：JWT 登录失败时，后续接口调用直接报 401，链路中断。
- 根因：虽然存在两种认证方式，但早期没有收敛出稳定的优先级和回退逻辑。
- 处理：把“JWT 优先、失败回退 API Key”明确为产品要求，并由配置驱动优先级。

### 问题 2：消息体中的嵌入认证仍然报 401

- 现象：测试页可以嵌入，但聊天消息中的看板嵌入会在 `/api/v1/me/roles/` 等接口上报 401。
- 根因：消息体内嵌使用到的 token 与真实嵌入认证链路并没有完全打通。
- 处理：需要把 guest token / auth token 的获取和传递路径收敛为可验证的统一方案。

### 问题 3：回退后仍然不可用

- 现象：日志显示“JWT 失败后回退”，但接口仍然不可用。
- 根因：API Key 可能未配置、未读取到，或者读取逻辑与预期不一致。
- 处理：回退路径本身也必须可观测、可验证，不能只在日志里宣称“已回退”。
