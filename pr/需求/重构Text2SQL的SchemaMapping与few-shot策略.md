# 重构 Text2SQL 的 Schema Mapping 与 few-shot 策略

- 需求编号：`需求-15`
- 首次提出：`2026-03-01`
- 最后有效确认：`2026-03-06`
- 时效状态：`有效`
- 实现状态：`进行中`
- 需求主线：`Text2SQL 语义映射`

## 当前定义

- 修正 Schema Mapping、字段候选暴露、few-shot exemplar 和相关 SQL 改写逻辑，避免 LLM 被技术字段名、错误别名和固定 SQL 模式误导。

## 用户要求

- 不要再让系统优先选 `*_id`、`*_key` 这类可读性差的技术字段。
- 先分析根因，不要直接写代码。
- 要真正改“从数据集字段里选出相关字段”的实现，而不是只改提示词表面。
- 要确认 few-shot、候选字段组织、字段名映射与 SQL 改写是如何共同导致问题的。

## 需求边界

- 本需求处理的是“字段为什么被选错、few-shot 为什么会误导、SQL 为什么被错误改写”。
- 空指针、脏数据、反序列化、字段名含空格导致的稳定性问题，属于 [提升语义查询链路稳定性](提升语义查询链路稳定性.md)。
- 模型关系、schema 默认值、下钻语义规则，属于 [梳理模型关联、字段命名与下钻语义规则](梳理模型关联字段命名与下钻语义规则.md)。

## 验收要点

- 候选字段更偏向用户真正需要的可读字段，而不是技术字段。
- 字段别名、展示名和物理列名在生成 SQL 时不再混乱。
- few-shot exemplar 相关边界 bug 被修正。
- 错误 SQL 不再因为二次改写或错误聚合而进一步恶化。

## 归属的 bug / 修复

### 问题 1：系统优先命中技术字段，而不是可读字段

- 现象：用户问“每个产品的销售额”时，系统容易优先选 `product_key`、`*_id` 之类字段。
- 根因：mapping 阶段暴露给 LLM 的候选字段过度偏向技术字段，用户可读字段反而不显眼。
- 处理：重构候选字段组织与 few-shot 引导，降低技术字段被优先选中的概率。

### 问题 2：字段别名和真实列名被反复混淆

- 现象：最终 SQL 经常出现对展示名、别名或指标名的错误聚合与错误替换。
- 根因：字段在 `bizName`、展示名、别名之间多轮改写，规则边界没有收敛。
- 处理：把 alias 剔除、展示名替换和字段回写逻辑收敛到可解释规则上。

### 问题 3：few-shot exemplar 存在边界 bug

- 现象：日志中出现 `Index -1 out of bounds for length 0` 等错误。
- 根因：exemplar 取样在空列表场景下没有做边界保护。
- 处理：补齐空列表与候选不足场景的边界处理。
