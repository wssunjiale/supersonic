# 建立 Superset 语义同步链路与离线验收运行器

- 需求编号：`需求-13`
- 首次提出：`2026-03-05`
- 最后有效确认：`2026-03-06`
- 时效状态：`有效`
- 实现状态：`进行中`
- 需求主线：`Superset 同步治理`

## 当前定义

- 建立一条可治理的 Superset 同步链路，包括数据集注册表、同步入口、失败重试、失败原因管理、手动同步入口，以及脱离完整 Web 启动的离线验收运行器。

## 用户要求

- 如果查询或画图过程产出了错误 SQL，就不要继续把错误定义同步到 Superset。
- 自动同步不能无差别乱刷，应优先处理未同步、失败或人工要求同步的数据集。
- 要区分“可自动恢复的失败”和“内部逻辑错误”，决定是否继续重试。
- 数据集管理页面应支持手动同步，并显示最近一次失败原因。
- 需要扫描并梳理现有代码入口、runner、配置和环境变量。
- 不要求每次都完整启动 Supersonic 才能做同步验收，应该能有独立运行器或测试程序。

## 需求边界

- 本需求处理“同步治理和验收机制”。
- “dataset 应该映射什么对象”属于 [按查询结果固化为 Superset 数据集](按查询结果固化为Superset数据集.md)。
- 图表类型选择、嵌入展示和聊天链路重构不属于本需求本体。

## 验收要点

- 错误 SQL 会在同步前被拦住。
- 同步失败有明确原因，并能区分是否自动重试。
- 管理页能手动触发同步并看到最近一次失败原因。
- 能说明同步入口、运行器入口和所需配置。

## 归属的 bug / 修复

### 问题 1：错误 SQL 不应继续同步

- 现象：画图失败或 SQL 明显错误时，系统仍可能继续生成 Superset dataset 定义。
- 根因：同步前缺少足够严格的有效性校验。
- 处理：把“同步前校验、失败即跳过”作为硬规则写进同步链路。

### 问题 2：失败后如何重试没有稳定规则

- 现象：用户明确要求 Superset 不在线等外部故障要继续重试，但内部逻辑错误不能无限重试。
- 根因：失败类型没有被清晰分层。
- 处理：将失败原因分成“可恢复”和“不可恢复”，分别决定自动重试或等待人工处理。

### 问题 3：同步兼容问题散落在多个会话里

- 现象：日期格式 422、`already exists`、sync 500 fatal error 等问题都出现在同步链路中，但历史上分散处理。
- 根因：缺少统一的同步治理主线来承接这些兼容问题。
- 处理：把这类 Superset sync API 兼容问题统一沉入本需求。
