# 基于 Headless 数据集同步链路重构聊天绘图与 Superset 嵌入方案

- 需求编号：`需求-21`
- 首次提出：`2026-03-11`
- 最后有效确认：`2026-03-11`
- 时效状态：`有效`
- 实现状态：`进行中`
- 需求主线：`Headless 主路径下的聊天绘图重构`

## 当前定义

- 以 Headless 已同步的数据集为唯一正确数据底座，重构聊天绘图链路、chart 创建链路和 Superset 嵌入方案；聊天结果仍由后端生成可嵌入 dashboard，前端只负责展示和交互。

## 用户要求

- Headless 元数据 / 数据集同步链路已经被明确指定为后续方案主路径。
- 聊天链路仍然要通过 Superset API 创建 chart。
- 前端仍然要通过 embedding SDK 嵌入 dashboard。
- 创建 chart 时仍然要经过“选择图表类型”和“配置图表参数”的过程。
- embed dashboard 时仍然需要能够提供 `form_data` 或等价运行时参数。
- 聊天消息中的资源需要长期保留，因为消息体会一直存在并需要反复查看。
- `viztype-online.json` 与 `form-data-schema` 需要被重新审视，不能假设已经够用。

## 需求边界

- 这不是简单替换展示层，而是聊天绘图整条主路径的重构。
- 图表类型选择和图表参数不会因为改走 Headless 数据底座而消失。
- 资源可以是持久资源，但默认不按“高复用共享资产”来设计。

## 验收要点

- 能说明一条以 Headless 已同步 dataset 为底座的完整聊天绘图时序。
- 后端仍能为消息体生成可嵌入的 dashboard。
- 图表类型选择与参数配置仍然存在，并与新主路径兼容。
- 能说清 `viztype-online.json` / `form-data-schema` 哪些可以直接复用、哪些还需补造。

## 归属的 bug / 修复

### 问题 1：当前聊天绘图链路和 Headless 同步链路是两套并行思路

- 现象：聊天绘图链路仍然会自行注册和同步 dataset，而用户已经明确认定 Headless 已同步数据集才是正确主路径。
- 根因：历史实现与新的架构认知还没有收敛到同一条主线。
- 处理：以 Headless 已同步 dataset 作为聊天绘图的唯一数据底座，重新定义后续 chart 与 embed 方案。

### 问题 2：切到新主路径后，选图和参数资产是否还够用

- 现象：用户已经让人重整 `viztype-online.json` 和 `form-data-schema`，但仍要求人工判断这些资产是否足以支撑新链路。
- 根因：现有资产可能只够覆盖局部 chart 创建，不一定够支撑完整重构。
- 处理：继续人工核定资产边界，并把不足之处沉成明确待补项。
