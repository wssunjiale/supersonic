# 提升语义查询链路稳定性

- 需求编号：`需求-17`
- 首次提出：`2026-02-26`
- 最后有效确认：`2026-03-10`
- 时效状态：`有效`
- 实现状态：`进行中`
- 需求主线：`语义查询稳定性`

## 当前定义

- 修复语义查询链路中的空指针、脏数据、字段名解析失败、请求体反序列化失败，以及 CLI / Web Bean 串用等稳定性问题。

## 用户要求

- `parseInfo` 极端重要，必须确保它存在或被正确回退。
- 先把根因研究清楚，不要拍脑袋修。
- 如果链路有问题，应立即暴露明确错误，而不是把业务问题降级成空指针或奇怪异常。

## 需求边界

- 本需求处理“链路为什么会炸、为什么会 NPE、为什么会解析失败”。
- 字段为什么会被选错、few-shot 为什么会误导，属于 [重构 Text2SQL 的 Schema Mapping 与 few-shot 策略](重构Text2SQL的SchemaMapping与few-shot策略.md)。

## 验收要点

- `parseInfo`、维度响应等关键对象缺失时，不再直接空指针。
- 特殊字段名、脏数据和错误请求体不会直接打爆主链路。
- CLI 专用 Bean 不会误注入到 Web 服务里。
- 出错时能给出可解释的失败原因。

## 归属的 bug / 修复

### 问题 1：`parseInfo` 或维度响应为空时直接空指针

- 现象：日志里多次出现 `SemanticParseInfo`、`dimensionResp` 相关空指针。
- 根因：关键对象读取失败后，后续链路没有做显式校验或可解释回退。
- 处理：把空值保护、显式报错和合理回退补到主链路中。

### 问题 2：字段名含空格、脏数据或错误请求体时解析链路直接炸掉

- 现象：`JSQLParserException`、`HttpMessageNotReadableException` 等异常反复出现。
- 根因：特殊字段名、脏数据和请求体结构没有被提前拦截或纠正。
- 处理：在 SQL 构建、请求体解析和展示名转换阶段分别补齐容错与校验。

### 问题 3：CLI 的 Noop Bean 被 Web 服务误用

- 现象：运行时出现 `UnsupportedOperationException: CLI noop`。
- 根因：CLI 用的 Noop Bean 没有和 Web 主实现彻底隔离。
- 处理：隔离 CLI 与 Web Bean 的注入范围，避免运行态串用。
